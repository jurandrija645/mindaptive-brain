# Mindaptive Brain

The central automation and intelligence platform for Mindaptive. It monitors the Gmail inbox, interprets incoming emails, and routes them to the right automation tools — currently posting real-time Slack notifications for booked meetings. Built to grow: a future UI, Telegram live chat, and an AI reasoning layer (Claude) are planned.

## Overview

Mindaptive Brain runs as a lightweight background service (FastAPI + Docker) on a DigitalOcean droplet. Every 30 seconds it polls the Gmail inbox using the Gmail History API — a highly efficient incremental approach that only fetches emails that arrived since the last check. Each new email is passed through a registry of automation tools. Tools decide independently whether they should act on the email, so multiple automations can trigger from a single message.

The first automations built are meeting notification tools: when a prospect books a Calendly or HubSpot meeting, a rich Slack message appears instantly in the configured channel.

## Architecture

```
Gmail Inbox
     │
     │  (polls every 30s via History API)
     ▼
┌─────────────┐
│ GmailPoller │  Incremental polling — only fetches new message IDs
└──────┬──────┘
       │  EmailMessage (normalized dataclass)
       ▼
┌────────────┐
│ Dispatcher │  The brain — routes emails, future home of LLM reasoning
└──────┬─────┘
       │
       ▼
┌──────────────┐
│ ToolRegistry │  Iterates all registered tools
└──────┬───────┘
       │  should_trigger()? → execute()
       ├──► CalendlyMeetingTool   → Slack notification
       ├──► HubspotMeetingTool    → Slack notification
       └──► EmailAlertTool        → Slack notification
                │
                ▼
           Slack Channel
```

**Key design principles:**
- Each tool is completely self-contained. It only needs to implement two methods: `should_trigger()` and `execute()`.
- Tools are decoupled — they do not know about each other.
- The `Dispatcher` is where future AI/LLM logic will live. Instead of simple rule-based routing, Claude will read the email and decide which tools to invoke and with what context.

## Current Automations

### Calendly Meeting Notifications (`calendly_meeting`)

**Trigger:** Any email from `notifications@calendly.com` or `calendar-confirmation@calendly.com`

**What it does:** Parses the person's name, meeting type, and scheduled time from the email, then posts a formatted Slack block message to the configured channel.

**Slack message includes:**
- Header: "📅 New Meeting Booked!" or "⏰ Meeting Reminder"
- Person name
- Meeting type (e.g. "30 Minute Meeting")
- Date and time with timezone
- Source: Calendly

---

### HubSpot Meeting Notifications (`hubspot_meeting_booked`)

**Trigger:** Email from `noreply@notifications.hubspot.com` with subject containing "prospect booked meeting"

**What it does:** Extracts the prospect name, email, and company from the notification body and posts to Slack.

---

### Email Alert (`email_alert`)

**Trigger:** Any email arriving from a specific watched address (currently `mindaptive@gmail.com`)

**What it does:** Posts a basic "New Email Received" notification to Slack with sender info and subject line. Useful for testing that the pipeline is alive.

## Local Setup

### Prerequisites

- Python 3.12+
- Docker & Docker Compose
- A Google Cloud project with Gmail API enabled
- A Slack bot with `chat:write` permission added to your workspace

### 1. Clone and configure environment

```bash
git clone <repo-url>
cd mindaptive-brain
cp .env.example .env
# Edit .env and fill in SLACK_BOT_TOKEN and SLACK_MEETING_CHANNEL
```

### 2. Set up Gmail credentials

1. Go to [Google Cloud Console](https://console.cloud.google.com/)
2. Create a project → Enable **Gmail API**
3. Go to **APIs & Services → Credentials → Create OAuth 2.0 Client ID** (Desktop app)
4. Download the JSON file and save it as `credentials/client_secret.json`

### 3. Run the Gmail OAuth flow (once)

```bash
cd backend
pip install -r requirements.txt
python setup_auth.py
```

This opens a browser window. Log in as `andrew@mindaptive.ai` (or your configured Gmail account). After approving, `credentials/token.json` is saved. **This file is your authentication credential — keep it secret.**

### 4. Run locally with Docker

```bash
docker compose up --build
```

Check the health endpoint:
```bash
curl http://localhost:8000/health
# {"status":"ok","version":"0.1.0"}
```

## Deploying to DigitalOcean

The app is deployed to a DigitalOcean Droplet and auto-deploys on every push to `main` via GitHub Actions.

### One-time server setup

```bash
# SSH into the droplet
ssh user@<droplet-ip>

# Clone the repo
git clone <repo-url> ~/Mindaptive-brain
cd ~/Mindaptive-brain

# Create credentials directory and upload secrets
mkdir -p credentials
# Upload credentials/token.json and credentials/client_secret.json via scp or nano

# Set up .env
cp .env.example .env
nano .env  # fill in SLACK_BOT_TOKEN etc.

# Start the service
docker compose up -d --build
```

### Continuous deployment

Any push to `main` triggers the GitHub Actions workflow (`.github/workflows/deploy.yml`), which SSHes into the droplet, pulls latest code, and rebuilds the Docker container. Required GitHub Secrets:

| Secret | Description |
|---|---|
| `SSH_HOST` | Droplet IP address |
| `SSH_USER` | SSH username (e.g. `root`) |
| `SSH_PRIVATE_KEY` | Private key for SSH access |

### Keeping credentials fresh

The Gmail `token.json` contains a refresh token that renews the access token automatically. However, **if the service is down for an extended period or Google revokes the token**, you will need to re-run `setup_auth.py` locally and re-upload `credentials/token.json` to the droplet.

## Adding a New Automation Tool

1. **Create a new file** in `backend/app/tools/`, e.g. `telegram_alert.py`

2. **Subclass `BaseTool`** and implement both methods:

```python
from app.models import EmailMessage
from app.tools.base import BaseTool

class TelegramAlertTool(BaseTool):
    name = "telegram_alert"

    async def should_trigger(self, email: EmailMessage) -> bool:
        # Return True if this email should trigger the tool
        return "urgent" in email.subject.lower()

    async def execute(self, email: EmailMessage) -> None:
        # Do something — send a message, call an API, etc.
        pass
```

3. **Register it** in `backend/app/main.py` inside `build_registry()`:

```python
from app.tools.telegram_alert import TelegramAlertTool

def build_registry(slack: SlackClient) -> ToolRegistry:
    registry = ToolRegistry()
    registry.register(CalendlyMeetingTool(slack=slack))
    registry.register(TelegramAlertTool(...))  # add here
    return registry
```

That's it. The tool will be automatically called for every new email.

## Troubleshooting

### Calendly/HubSpot notifications not arriving in Slack

Work through these checks in order:

**1. Verify the container is running on the droplet**
```bash
ssh user@<droplet-ip>
cd ~/Mindaptive-brain
docker compose ps
```
If the container is not `Up (healthy)`, check logs (step 2).

**2. Check the application logs**
```bash
docker compose logs -f brain
```
Look for:
- `Gmail poller started at historyId=...` — confirms the app started
- `Tool 'calendly_meeting' triggered for email ...` — confirms a tool ran
- `Slack API error: ...` — Slack token or channel issue
- `Poll error: ...` — Gmail API issue

**3. Verify the Gmail token is valid**

If you see `invalid_grant` or `401` errors in the logs, the token has expired. Fix:
```bash
# Locally:
cd backend && python setup_auth.py
# Then upload the new token.json to the droplet:
scp credentials/token.json user@<droplet-ip>:~/Mindaptive-brain/credentials/token.json
# Restart:
ssh user@<droplet-ip> "cd ~/Mindaptive-brain && docker compose restart"
```

**4. Confirm which email address Calendly sends to**

Calendly sends notifications to the email address registered on your Calendly account — make sure that is the same Gmail inbox being polled (configured as `GMAIL_USER` in `.env`). If your Calendly account is tied to a different address, the poller will never see those emails.

**5. Check the Calendly sender address**

Calendly sometimes uses different "From" addresses depending on the notification type (booking confirmation vs. reminder vs. cancellation). Check the raw email headers in Gmail (open the email → three-dot menu → "Show original") and confirm the `From:` field contains `calendly.com`. The tool currently matches on `notifications@calendly.com` and `calendar-confirmation@calendly.com`.

**6. Verify the Slack bot is in the channel**

The bot must be **invited** to the channel set in `SLACK_MEETING_CHANNEL`. In Slack: open the channel → click the channel name → **Integrations** → **Add apps** → find your bot. If the bot is not in the channel, you'll see `not_in_channel` in the logs.

**7. Force a test email**

Send an email from `mindaptive@gmail.com` to `andrew@mindaptive.ai`. Within 30 seconds you should see a "New Email Received" notification in Slack (from the `EmailAlertTool`). If this works but Calendly doesn't, the pipeline is alive and the issue is specific to how the Calendly email is parsed.

## Roadmap

The project is being built incrementally. Current focus and planned additions:

### Now (v0.1 — complete)
- Gmail polling via History API
- Calendly meeting booked/reminder → Slack
- HubSpot prospect booked meeting → Slack
- Docker deployment with GitHub Actions auto-deploy

### Next
- **AI reasoning layer** — Claude reads each email and decides how to route it, instead of pattern-matching rules. The `Dispatcher` is already the right place to add this.
- **Telegram integration** — Live chat interface so the Mindaptive team can interact with the brain directly from Telegram. Inbound messages trigger automations; the AI can reply.
- **Web UI** — A dashboard to view notification history, configure tools, and see system health.

### Future automations (examples)
- Auto-reply or follow-up email sequences triggered by lead activity
- CRM updates (e.g. create a HubSpot contact when a new prospect books)
- Lead scoring and prioritization via AI
- Daily/weekly digest reports sent to Slack
- Integration with calendar APIs to check availability

## Environment Variables Reference

| Variable | Required | Default | Description |
|---|---|---|---|
| `SLACK_BOT_TOKEN` | Yes | — | Slack bot OAuth token (`xoxb-...`) |
| `SLACK_MEETING_CHANNEL` | No | `danteov-kanal` | Slack channel name (without #) |
| `GMAIL_TOKEN_PATH` | No | `credentials/token.json` | Path to OAuth token file |

| `GMAIL_USER` | No | `me` | Gmail user ID (`me` = authenticated user) |
| `POLL_INTERVAL_SECONDS` | No | `30` | How often to check Gmail (in seconds) |
| `LOG_LEVEL` | No | `INFO` | Logging verbosity (`DEBUG`, `INFO`, `WARNING`) |
| `ENVIRONMENT` | No | `production` | Environment label |
